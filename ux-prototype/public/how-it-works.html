System Architecture

The parts of BARTOK, how they talk to each other, and where each one runs. The consumer sees a plain chat app; underneath, every reply is attested by TLSNotary and every payment settles as a private note on Miden testnet, co-signed by an application Guardian.

P2P verifiable-inference marketplace · Miden testnet · updated 2026-07-08

Three hosting zones

Every component lives in one of three places. Colour follows the host throughout this page.

01The components

Rita’s app

The chat UI and a real Miden wallet, running as WebAssembly in the browser. The wallet is a Guardian multisig (one Falcon key kept in localStorage + the Guardian’s co-signature). None of the words wallet / note / chain ever appear to the user.
host: browserserved by VercelJS + Miden SDK WASM

Vercel

Static hosting for the app bundle only (the 15 MB of HTML/JS/WASM). It runs no server logic; it just delivers the page with the cross-origin-isolation headers the WASM needs.
host: Vercel edgebartok-ten.vercel.app

Bridge

The Node server and the brain of the backend: manages sessions, routes tiers to models, orchestrates the zkTLS pipeline, drives settlement by spawning the Rust bins, runs the credits funnel, and streams João’s dashboard feed.
host: VPSNode :8787systemd: bartok-bridge

Bartok-Guardian

The application co-signer (a fork of OpenZeppelin’s Guardian). It holds the second key of every multisig and ACKs each value-moving transaction, then watches the chain to canonicalize account deltas. This is what makes the wallets non-custodial and recoverable.
host: VPSRust :3300 / :50052systemd: bartok-guardian

Rust bins

Short-lived processes the bridge spawns: build_escrow (emits the private escrow as transaction bytes), settle_session (operator multisig consumes the escrow and splits it), fund_buyer (credit mints), joao_sweep (seller collects payment).
host: VPSRust release binscwd miden/integration

zkTLS pipeline

TLSNotary examples the bridge runs per reply: a prover + notary attest the OpenRouter TLS session; selective disclosure reveals only model + usage; an oracle verifies the notary key and computes the charge = tokens × tier price.
host: VPSRust (tlsn)→ OpenRouter

Disk state

On the VPS filesystem: the Miden client SQLite stores + keystores, the Guardian’s Falcon/ECDSA ACK keys, the notary key, and the bridge’s sessions/users/refunds JSON. All gitignored, none web-served.
host: VPSSQLite + key files

Miden testnet + prover + OpenRouter

The public services BARTOK builds on: the testnet RPC node (the chain itself) + midenscan explorer, the remote STARK prover, and OpenRouter (the real LLMs). None are ours; we point at them.
host: publictestnet3rd-party APIs

02On Guardian: today vs what it enables

A fair question when you see the diagram: in today’s core loop, Guardian’s co-signature isn’t yet doing security work. Here is the honest split.

TODAYEarning its keep
  • Recoverable, non-custodial wallets. Rita’s key lives in the browser. A single-key wallet means a cleared browser loses her funds forever; a multisig with Guardian as the second signer makes that survivable. Essential for a non-technical phone user (and the recovery UX is only partly wired so far).
  • Co-signs every value-moving tx — but only checks state-consistency (the delta cleanly extends the account’s canonical chain), not business policy. As a security control it is a rail, not yet a guard.
ENABLESThe foundation for
  • Policy-ACK (the trust endgame): Guardian validates the charge against the oracle attestation before co-signing — a hook on the co-signature that already exists.
  • Spend limits & allowlists, session keys (approve a whole chat once), sponsored / pre-funded accounts, social recovery, and key rotation without moving funds.
Honest one-liner — today Guardian buys recoverable non-custodial wallets; its real value is being the policy-and-recovery layer every trust-minimizing feature plugs into next. The pay-per-token + verified-model loop could ship with plain single-key wallets, at the cost of “clear your browser, lose your money” and no path to the policy-ACK endgame.

03How a barter flows

Four sequences cover the whole lifecycle. Actor colour = where the step runs: browser backend/VPS public.

SETUPFirst visit — the invisible wallet▸
  1. browser The app boots the Miden SDK (WASM) and creates a Guardian multisig account: one Falcon key generated and saved in localStorage.
  2. browser → Guardian registers the account with Bartok-Guardian (its co-signer key commitment). The wallet now survives cleared storage via Guardian recovery.
  3. browser → testnet syncs account state directly from the RPC node. No funds yet.
REDEEMTop-up with a code → real BarŦoks▸
  1. browser → Bridge POSTs the code (e.g. ILOVEBARTOK) to /api/credits/redeem.
  2. Bridge spawns fund_buyer, which mints a private note from the faucet to the user’s account (charge attested purely in integer base units).
  3. Bridge → browser returns the mint note as bytes; the browser consumes it via a Guardian proposal (propose → Guardian co-signs → execute). Balance appears.
CHATAsk a question — paid + proven▸
  1. browser → Bridge /api/session/start; the bridge runs build_escrow to emit a private BartokSettlement note as tx bytes.
  2. browser funds that escrow as a Guardian custom proposal (co-signed) — a small hold set aside on-chain. This is the “~1–2 min setting up” wait.
  3. browser → Bridge /api/chat. The bridge runs the reply inside TLSNotary: zkTLS → OpenRouter, notary attests the session.
  4. oracle verifies the notary key + selective disclosure, derives charge = tokens × price, and the reply streams back with a ✓ verified chip. History rides inside the 4 KiB proof channel.
SETTLEEnd barter — private split + refund▸
  1. browser → Bridge /api/session/end; the bridge runs settle_session.
  2. operator multisig consumes the escrow as an unauthenticated input note (full bytes over the wire, nothing published) and splits it: a P2ID to João’s multisig + a private P2ID refund to the buyer. Guardian co-signs.
  3. Bridge → browser hands back the refund NoteFile; the browser consumes it (Guardian proposal) and the balance ticks up. A background joao_sweep collects João’s payment.
  4. Guardian → testnet canonicalizes all three accounts’ deltas as the settlement tx confirms. Receipt shows a testnet record link.

04Trust & privacy, honestly

Guardian co-signature — every value-moving transaction requires Bartok-Guardian’s ACK over that exact tx summary, and Guardian only co-signs deltas that consistently extend an account’s canonical chain. Business-policy validation (charge matches the attestation) is not enforced yet: infrastructure in place, policy hook deferred.
Private notes — escrow, refund and mint are all private; midenscan shows commitments only. Settlement consumes the escrow as unauthenticated bytes so nothing is published.
Attestation, not zk — reply provenance is TLSNotary MPC attestation + selective disclosure of model/usage, not a zk circuit. Today the notary is operator-run; independent/threshold notaries + a note script that verifies the charge via FPI are the designed closure path.
Everything chain-touching is testnet — no real value moves. MockChain regression tests cover the settlement contract.

Where it all runs today: the app bundle on Vercel; the bridge, Bartok-Guardian, the Rust bins, the zkTLS pipeline and all keys/state on one DigitalOcean VPS (three systemd services, ngrok outbound tunnel, SSH-only firewall); the chain, prover and LLMs are public services we call. The browser reaches the backend through the ngrok static domain, and reaches testnet directly for wallet operations.
